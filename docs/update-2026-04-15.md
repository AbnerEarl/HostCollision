## 一、新增功能

### 1. 方案六：万能响应IP检测（Catch-All Detection）

**问题背景**：某些 IP 上运行着默认虚拟主机、通配符配置或对任何 Host 头都返回响应的服务器，无论传入什么 Host 头都会"碰撞成功"，产生大量无效结果。

**解决方案**：在碰撞过程中实时统计每个 IP+协议 维度的碰撞成功数量，当超过阈值时判定为"万能响应"IP，回溯标记该 IP 的所有碰撞结果为无效，并提前终止该 IP 剩余 Host 的碰撞。

**核心逻辑**：
- 实时统计每个 IP+协议 的碰撞成功计数
- 超过阈值（默认 10）时判定为万能响应 IP
- 使用**标记式删除**（`Invalid` 字段）而非物理删除，避免与 `RunWithCallback` 的回调索引产生并发冲突
- 重试队列中也会继续检测万能响应
- `RunWithOptions` 和 `RunWithCallback` 中自动过滤 `Invalid` 结果

**新增参数**：
| 参数 | 命令行 | 默认值 | 说明 |
|------|--------|--------|------|
| `EnableCatchAllDetection` | `-catchall` | `true` | 是否启用万能响应IP检测 |
| `CatchAllThreshold` | `-cat` | `10` | 万能响应IP判定阈值 |

**日志输出示例**：
```
[万能响应IP检测] 协议:http://, IP:1.1.1.1 碰撞成功 10 个Host(阈值10), 判定为万能响应IP, 已清除 10 条结果, 跳过剩余 890 个Host
```

**涉及文件**：
- `pkg/config/config.go` — `OptimizationConfig` 新增 `EnableCatchAllDetection` 和 `CatchAllThreshold` 字段
- `pkg/collision/collision.go` — `CollisionResult` 新增 `Invalid` 标记字段；`processIPProtocol` 添加实时计数和阈值检测；新增 `removeCatchAllResults` 方法
- `hostcollision.go` — `Options` 和 `Result` 结构体更新；`RunWithOptions` / `RunWithCallback` 过滤无效结果
- `cmd/hostcollision/main.go` — 新增 `-catchall` 和 `-cat` 命令行参数
- `resource/config.yml` — 新增配置项

### 2. SimHash 响应体指纹（相似内容聚合）

**问题背景**：碰撞结果中可能存在大量相似的响应页面（如同一个默认页面、同一个 WAF 拦截页面），需要一种方式快速识别和聚合相似内容。

**解决方案**：在碰撞成功时计算响应体的 64 位 SimHash 指纹（局部敏感哈希），用户可通过海明距离判断内容相似度。

**SimHash 算法流程**：
1. 将文本按空格/标点分词为 token 列表（支持中文 bigram）
2. 对每个 token 计算 FNV-1a 64 位 hash
3. 构建 64 维加权向量：hash 的每一位为 1 则 +1，为 0 则 -1
4. 向量每一维正数映射为 1，非正数映射为 0，组成 64 位指纹

**海明距离判断**：
- 距离 = 0：内容完全相同
- 距离 ≤ 3：内容高度相似
- 距离 > 3：内容差异较大

**新增字段**：
- `CollisionResult.BodySimhash` — 内部结构体
- `Result.BodySimhash` — 公开结构体
- CSV 输出新增 `body_simhash` 列
- TXT/日志输出新增 `body_simhash` 字段

**新增函数**：
| 函数 | 说明 |
|------|------|
| `simhash(text)` | 核心算法：分词 → FNV hash → 64 维加权向量 → 64 位指纹 |
| `tokenize(text)` | 分词器：支持英文和中文（bigram 二元组） |
| `isTokenChar(r)` | 判断有效 token 字符（字母、数字、CJK 汉字、日文假名、韩文音节） |
| `hasCJK(runes)` | 检测是否包含 CJK 字符 |
| `SimhashDistance(a, b)` | 计算两个 SimHash 的海明距离（公开方法，供外部调用） |

**涉及文件**：
- `pkg/collision/collision.go` — 新增 SimHash 算法实现和结构体字段
- `hostcollision.go` — `Result` 结构体新增 `BodySimhash` 字段

### 3. 日志过滤：抑制 Go 标准库无关日志

**问题背景**：Go 标准库 `net/http` 的 `Transport` 在空闲连接上收到未请求的响应时，会通过 `log.Printf` 输出 `Unsolicited response received on idle HTTP channel` 日志，污染标准输出。

**解决方案**：创建 `filterWriter` 结构体实现 `io.Writer` 接口，通过 `log.SetOutput()` 将 Go 标准库的 `log` 输出重定向到过滤 writer，静默丢弃包含特定关键词的日志。

**实现细节**：
- 使用 `init()` 函数自动初始化，无需手动调用
- 使用 `sync.Once` 确保只初始化一次
- 仅过滤 `Unsolicited response received on idle HTTP channel` 关键词
- 其他正常日志不受影响

**涉及文件**：
- `pkg/httpclient/httpclient.go` — 新增 `filterWriter`、`SuppressUnsolicitedResponseLog()` 和 `init()` 函数

## 二、Bug 修复

### 1. Title 字段在 301/302 重定向时为空

**问题描述**：通过库模式调用时，保存到数据库的很多数据 `title` 为空，但日志打印时 `title` 是存在的。

**根因分析**：`HttpCustomRequest.Title()` 方法调用 `r.AppBody()` 来提取标题，但当 HTTP 响应包含 `Location` 头（301/302 重定向）时，`AppBody()` 返回的是 URL 字符串（如 `"http://xxx.com"`）而非原始 HTML，导致正则 `<title>(.*?)</title>` 匹配不到任何内容。

**修复方案**：修改 `Title()` 方法，**优先从 `r.Body`（原始响应体）提取 title**，提取不到时再回退到 `r.AppBody()`。

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 正常响应（无 Location） | ✅ 正常 | ✅ 正常 |
| 重定向响应（有 Location） | ❌ Title 为空 | ✅ 从原始 Body 提取 |

**涉及文件**：
- `pkg/httpclient/httpclient.go` — 修改 `Title()` 方法
