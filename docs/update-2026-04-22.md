# 更新日志 - 2026-04-22

## 概述

本次更新主要围绕 **碰撞结果质量提升** 和 **数据完整性保障** 两个方向，修复了 Title 中文乱码问题、碰撞结果重复问题，以及 CLI 入口中万能响应IP无效结果未过滤的问题。

## 一、Bug 修复

### 1. Title 中文乱码修复（编码检测与自动转换）

**问题描述**：通过库模式或 CLI 调用时，部分中文网站的 Title 显示为乱码，例如 "邮箱用户登录-阿里邮箱提供技术支持" 显示为 `閭鐢ㄦ埛鐧诲綍-闃块噷閭鎻愪緵鎶鏈敮鎸`。

**根因分析**：服务器返回的 HTML 页面使用了 GBK/GB2312 等非 UTF-8 编码（中文网站常见），但 `SendHTTPGetRequest` 中直接使用 `string(body)` 将字节转为字符串。Go 默认按 UTF-8 解码，导致非 UTF-8 编码的中文内容变成乱码。项目中此前完全没有编码检测和转换逻辑。

**修复方案**：在 HTTP 响应体转字符串时，新增编码检测与自动转换逻辑，将非 UTF-8 编码的响应体自动转换为 UTF-8。

**新增依赖**：
| 依赖 | 版本 | 用途 |
|------|------|------|
| `golang.org/x/net` | v0.53.0 | `html/charset` 包：综合 Content-Type header 和 HTML meta 标签检测编码 |
| `golang.org/x/text` | v0.36.0 | `transform` 包：执行编码转换 |

**新增函数**：
| 函数 | 文件 | 说明 |
|------|------|------|
| `detectAndConvertToUTF8(body, contentType)` | `pkg/httpclient/httpclient.go` | 核心函数：检测响应体编码并转换为 UTF-8 |
| `isValidUTF8(data)` | `pkg/httpclient/httpclient.go` | 辅助函数：通过统计无效 UTF-8 字节比例判断编码合法性 |

**编码检测优先级**：
1. Content-Type header 中的 charset 声明
2. HTML `<meta charset>` 标签
3. 自动检测（`charset.DetermineEncoding`）
4. 如果无法确定编码但内容是合法 UTF-8，直接返回原始内容
5. 如果转换失败，回退返回原始内容（不会因编码问题导致程序崩溃）

**修改位置**：`SendHTTPGetRequest` 函数中，将 `string(body)` 替换为 `detectAndConvertToUTF8(body, contentTypeHeader)`。

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| UTF-8 编码页面 | ✅ 正常 | ✅ 正常 |
| GBK/GB2312 编码页面 | ❌ 中文乱码 | ✅ 自动转换为 UTF-8 |
| 其他编码（如 Shift_JIS、EUC-KR 等） | ❌ 乱码 | ✅ 自动转换为 UTF-8 |
| 编码检测失败 | - | ✅ 回退返回原始内容 |

**涉及文件**：
- `pkg/httpclient/httpclient.go` — 新增编码检测转换函数，修改 `SendHTTPGetRequest`
- `go.mod` — 新增 `golang.org/x/net` 和 `golang.org/x/text` 依赖
- `go.sum` — 更新依赖校验

### 2. 碰撞结果重复修复（全局去重机制）

**问题描述**：碰撞结果中出现相同 Protocol+IP+Host 的重复记录，例如同一个 `http:// | 106.11.237.147 | cas.ctripcorp.com` 出现多次（Title 不同）。按业务逻辑，Protocol+IP+Host 三者应构成唯一资产。

**根因分析**：碰撞核心逻辑中**没有对 Protocol+IP+Host 做去重检查**。同一个 Host 可能在以下场景被多次碰撞成功：
1. 正式碰撞循环中碰撞成功一次
2. 被 WAF 拦截后加入重试队列，重试时又碰撞成功一次
3. 由于负载均衡后端不同，同一 Host 的两次请求返回不同内容（如一次中文页面、一次英文页面），两次都通过了碰撞检查

**修复方案**：在碰撞成功保存结果时，新增全局去重机制。

**实现细节**：
- 在 `Worker` 结构体中新增 `resultDedup map[string]struct{}` 字段，作为全局共享的去重集合
- 修改 `NewWorker` 函数签名，新增 `resultDedup` 参数，所有 Worker 共享同一个去重实例
- 去重键格式：`Protocol|IP|Host`（如 `http://|1.2.3.4|example.com`）
- 去重检查在 `resultsMu` 互斥锁保护下执行，确保线程安全
- 已存在的重复结果直接跳过，不写入结果列表

**去重逻辑位置**：`collision()` 方法中，在 `*w.results = append(*w.results, result)` 之前检查。

**所有入口函数均已更新**：

| 入口函数 | 文件 | 创建 `resultDedup` | 传入 `NewWorker` |
|----------|------|:--:|:--:|
| `RunWithOptions` | `hostcollision.go` | ✅ | ✅ |
| `RunWithCallback` | `hostcollision.go` | ✅ | ✅ |
| `main()` | `cmd/hostcollision/main.go` | ✅ | ✅ |

**涉及文件**：
- `pkg/collision/collision.go` — `Worker` 结构体新增 `resultDedup` 字段；`NewWorker` 新增参数；`collision()` 方法添加去重检查
- `hostcollision.go` — `RunWithOptions` 和 `RunWithCallback` 创建共享去重 map 并传入 Worker
- `cmd/hostcollision/main.go` — `main()` 创建共享去重 map 并传入 Worker

### 3. CLI 入口万能响应IP无效结果未过滤

**问题描述**：`cmd/hostcollision/main.go` 中的 CSV/TXT 文件写入和最终控制台输出，没有检查 `Invalid` 标记，导致被万能响应IP检测标记为无效的结果仍然会出现在输出中。

**说明**：`RunWithOptions` 和 `RunWithCallback`（库模式入口）已在上一版本（2026-04-15）中正确过滤了 `Invalid` 结果，但 CLI 入口 `main()` 遗漏了此过滤逻辑。

**修复方案**：在 `main.go` 的 3 处结果输出位置添加 `Invalid` 过滤：

| 位置 | 修复内容 |
|------|---------|
| `ticker.C` 分支 — CSV 写入 | 添加 `if results[i].Invalid { continue }` |
| `ticker.C` 分支 — TXT 写入 | 添加 `if results[i].Invalid { continue }` |
| `done` 分支 — CSV/TXT 最终写入 | 添加 `if !results[i].Invalid` 检查 |
| `done` 分支 — 控制台最终输出 | 添加 `if !r.Invalid` 检查，改用 `validCount` 统计有效结果数 |

**涉及文件**：
- `cmd/hostcollision/main.go` — 4 处添加 `Invalid` 过滤

## 四、碰撞流程更新

```
碰撞流程（含本次更新）:
IP + Host 输入
  → [已有] Host 清洗（格式校验、去重、TLD校验、异常检测）
  → [已有] TLS 证书 SAN 提取（P0 优先级标记）
  → [已有] DNS 反向筛选（/16 或 /24 网段匹配）
  → [已有] 响应快速排除（采样 50 个 Host）
  → [已有] HEAD 预筛选
  → [已有] FNV-1a 指纹缓存
  → [已有] 自适应分阶段采样
  → [已有] 万能响应IP检测（阈值 10，超过则标记无效并跳过）
  → [已有] SimHash 指纹计算（碰撞成功时计算）
  → [新增] 响应体编码检测与 UTF-8 自动转换（修复中文 Title 乱码）
  → [新增] Protocol+IP+Host 全局去重（避免重复结果）
  → [修复] CLI 入口过滤万能响应IP无效结果
  → 输出结果
```
