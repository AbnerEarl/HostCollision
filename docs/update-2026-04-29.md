# 更新日志 - 2026-04-29

## 概述

本次更新主要围绕 **编码检测精度提升**、**隐形资产过滤逻辑重构**、**重定向跟随支持** 和 **SimHash 去重增强** 四个方向，修复了 UTF-8 中文页面被误判为 GBK 导致的 Title 乱码问题，重写了直接请求 Host 的过滤逻辑（引入 DNS 解析验证），为碰撞请求增加了最多 3 次重定向跟随，并增强了 SimHash 比对用于直接请求 Host 的相似度过滤。

## 一、Bug 修复

### 1. UTF-8 中文页面 Title 乱码修复（编码检测逻辑重写）

**问题描述**：部分 UTF-8 编码的中文页面（如 `www.suanya.com`），Title 显示为乱码 `鐏�溅绁ㄦ煡璇�鐏�溅绁ㄨ�璐�綉绔�12306銆愭櫤琛岀伀杞︾エ銆�`，正确内容应为 `火车票查询_火车票订购网站12306【智行火车票】`。

**根因分析**：

`detectAndConvertToUTF8` 函数在 `charset.DetermineEncoding` 返回 `utf-8` 且 `certain=false` 时（Content-Type 无 charset 声明，仅 HTML meta 标签声明了 `charset=utf-8`），进入了"不确定是 UTF-8"的分支。该分支的旧逻辑是：尝试 GBK 解码，比较 CJK 字符数，选择更多的那个。

问题在于 **GBK 解码 UTF-8 中文内容时会产生 CJK 字符数"膨胀"**：

| 解码方式 | CJK 字符数 | 原因 |
|---------|-----------|------|
| UTF-8（正确） | 875 | 原始中文字符 |
| GBK（错误） | 1205 | 3 字节 UTF-8 中文序列被 GBK 拆成 2 个字符，部分恰好落在 CJK 范围 |

由于 `1205 > 875`，函数错误地选择了 GBK 解码结果，导致乱码。

此外，在非 UTF-8 分支中也存在类似问题：当 `charset.DetermineEncoding` 返回 `windows-1252`（Latin-1）时，对 GBK 编码的内容进行 Latin-1 解码，`countCJKChars(decoded)=0 >= countCJKChars(body)=0` 条件成立（0 >= 0），返回了错误的 Latin-1 解码结果（如 `ËÑºü³©ÓÎ`）。

**修复方案**：重写 `detectAndConvertToUTF8` 函数的编码检测逻辑，增加三层防护：

**层级1 — `hasUTF8MetaCharset()` 检查（新增函数）**：
```go
func hasUTF8MetaCharset(body []byte) bool {
    // 检查前 1024 字节中是否有 <meta charset="utf-8"> 声明
    // 支持: charset="utf-8"、charset='utf-8'、charset=utf-8
}
```
如果 HTML meta 标签明确声明了 `charset=utf-8`，直接信任 UTF-8，不再尝试 GBK 解码。

**层级2 — CJK 字符数 > 0 检查**：
如果原始 UTF-8 内容已经包含 CJK 字符（`utf8CJK > 0`），说明它确实是 UTF-8 编码的中文内容，不应该再尝试 GBK 解码。

**层级3 — 仅当 UTF-8 CJK=0 时才尝试 GBK**：
只有原始内容没有 CJK 字符但包含高字节（`>= 0x80`）时，才尝试 GBK 解码。这才是真正的 GBK 内容被误判为 UTF-8 的场景。

**非 UTF-8 分支增强**：
当检测到的编码（如 `windows-1252`）解码后 CJK 字符数为 0，且原始内容包含高字节时，额外尝试 GBK 解码作为备选。

**新增辅助函数**：

| 函数 | 功能 |
|------|------|
| `hasUTF8MetaCharset(body []byte) bool` | 检查 HTML meta 标签是否声明了 UTF-8 编码 |
| `tryGBKDecode(body []byte) string` | 尝试 GBK 解码，失败返回空字符串 |

**新增依赖**：
- `unicode/utf8` — UTF-8 字符串合法性验证
- `golang.org/x/text/encoding/simplifiedchinese` — GBK 编码器/解码器

**验证结果**：

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| UTF-8 中文页面（meta charset=utf-8） | ❌ `鐏�溅绁ㄦ煡璇�...` | ✅ `火车票查询_火车票订购网站12306【智行火车票】` |
| GBK 页面无 charset 声明 | ❌ `ËÑºü³©ÓÎ` | ✅ `搜狐畅游` |
| GBK 页面有 meta charset=gbk | ✅ 正常 | ✅ 正常 |
| GBK 页面 Content-Type 声明 charset | ✅ 正常 | ✅ 正常 |
| 纯 UTF-8 英文页面 | ✅ 正常 | ✅ 正常 |
| UTF-8 中文页面（Content-Type 声明） | ✅ 正常 | ✅ 正常 |

**涉及文件**：
- `pkg/httpclient/httpclient.go` — 重写 `detectAndConvertToUTF8` 函数；新增 `hasUTF8MetaCharset`、`tryGBKDecode` 函数；新增 `unicode/utf8`、`simplifiedchinese` 包导入

## 二、功能优化

### 1. 直接请求 Host 过滤逻辑重构（引入 DNS 解析验证）

**问题背景**：之前的"匹配失败-5"过滤逻辑只比对碰撞请求与直接请求 Host 的**响应内容**，没有检查 Host 的 DNS 是否解析到了碰撞的目标 IP。这导致了一个严重的误过滤场景：

```
nephele.ctrip.com 的 DNS → 解析到 IP-A（某个 CDN 节点）
碰撞目标 IP: 23.45.207.83（另一个 CDN 节点）

两个 CDN 节点部署了相同的服务 → 响应内容高度相似
→ 被"匹配失败-5(响应与直接请求Host高度相似)"错误过滤
```

但这**恰恰是隐形资产**——`nephele.ctrip.com` 的 DNS 并不指向 `23.45.207.83`，说明 `23.45.207.83` 上部署了该 Host 的服务但未公开。

**修复方案**：重写"匹配失败-5"逻辑，改为**先检查 DNS 解析，再决定是否过滤**：

```
1. net.LookupHost(host) 获取 Host 的 DNS 解析结果
2. 如果解析到的 IP 包含碰撞目标 IP
   → 直接过滤（DNS 直接指向目标 IP，非隐形资产）
3. 如果解析到的 IP 不包含碰撞目标 IP
   → 条件1：状态码 + 内容长度 + Body 完全一致 → 过滤（CDN 分发相同内容）
   → 条件2：状态码相同 + SimHash 海明距离 ≤ 3 → 过滤（动态内容但本质相同）
   → 其他情况 → 保留（这是隐形资产）
4. 如果 DNS 解析失败 → 不过滤（保留碰撞结果）
```

**新增代码逻辑**：
```go
hostIPs, dnsErr := net.LookupHost(host)
if dnsErr == nil && len(hostIPs) > 0 {
    // 检查 DNS 解析结果是否包含碰撞目标 IP
    hostResolvesToTargetIP := false
    for _, resolvedIP := range hostIPs {
        if resolvedIP == ip {
            hostResolvesToTargetIP = true
            break
        }
    }
    if hostResolvesToTargetIP {
        return collisionResultFailed // DNS 直接指向目标 IP
    }
    // DNS 指向不同 IP，使用严格比对 + SimHash 相似度比对
    ...
}
```

**日志输出示例**：
```
协议:http://, ip:1.2.3.4, host:example.com 匹配失败-5(Host的DNS直接解析到目标IP,非隐形资产)
协议:http://, ip:1.2.3.4, host:example.com 匹配失败-5(响应与直接请求Host结果完全一致,非隐形资产)
协议:http://, ip:1.2.3.4, host:example.com 匹配失败-5(响应与直接请求Host结果高度相似,SimHash海明距离≤3,非隐形资产)
```

**涉及文件**：
- `pkg/collision/collision.go` — 重写 `collision()` 方法中的"匹配失败-5"逻辑；新增 `net` 包导入

### 2. 碰撞请求支持重定向跟随（最多 3 次）

**问题背景**：`SendHTTPGetRequest` 中 `CheckRedirect` 设置为 `return http.ErrUseLastResponse`，完全不跟随重定向。这意味着碰撞请求得到 301/302 时，只获取到重定向页面本身（通常是很小的 HTML 或空 body），无法看到最终目标页面的真实内容，导致比对逻辑基于不完整的信息做判断。

**修复方案**：`SendHTTPGetRequest` 改为最多跟随 **3 次**重定向，并在跟随时保持原始 Host 头：

```go
const maxRedirects = 3

CheckRedirect: func(req *http.Request, via []*http.Request) error {
    if len(via) >= maxRedirects {
        return http.ErrUseLastResponse  // 超过 3 次，停止跟随
    }
    // 跟随重定向时保持原始 Host 头
    if len(via) > 0 && via[0].Host != "" {
        req.Host = via[0].Host
    }
    return nil
},
```

**关键设计决策**：跟随重定向时**保持原始 Host 头**。这对 Host 碰撞场景至关重要——如果服务器返回 `301 → https://example.com/`，跟随时仍然需要使用碰撞的 Host 头，否则重定向后的请求会变成直接访问 IP。

**不修改的函数**：

| 函数 | 重定向策略 | 原因 |
|------|-----------|------|
| `SendHTTPGetRequestQuick` | 不跟随 | IP 预检测，只需看第一个响应 |
| `SendHTTPHeadRequest` | 不跟随 | HEAD 预筛选，只需看响应头 |

**涉及文件**：
- `pkg/httpclient/httpclient.go` — `SendHTTPGetRequest` 函数修改 `CheckRedirect` 逻辑

### 3. SimHash 直接请求 Host 相似度比对增强

**问题背景**：在直接请求 Host 的比对中，原有逻辑仅支持严格比对（状态码 + 内容长度 + Body 完全一致）。但实际场景中，同一页面可能包含动态内容（时间戳、token、session ID 等），导致两次请求的 Body 不完全一致，但本质是同一页面。

**修复方案**：在严格比对之后，增加 SimHash 相似度比对作为补充过滤条件：

```go
// 条件2：状态码相同，Body 不完全一致（可能有动态内容）
// 使用 SimHash 相似度比对，海明距离 ≤ 3 视为同一页面
if hostDirectReq.StatusCode == newRequest.StatusCode {
    collisionSimhash := simhash(newRequest.FilteredPageContent())
    directSimhash := simhash(hostDirectReq.FilteredPageContent())
    if SimhashDistance(collisionSimhash, directSimhash) <= 3 {
        return collisionResultFailed
    }
}
```

| 比对条件 | 过滤策略 | 适用场景 |
|---------|---------|---------|
| 状态码 + 长度 + Body **完全一致** | 直接过滤 | 静态页面、CDN 分发 |
| 状态码相同 + SimHash 海明距离 **≤ 3** | 过滤 | 页面有动态内容但本质相同 |

**涉及文件**：
- `pkg/collision/collision.go` — `collision()` 方法中"匹配失败-5"逻辑增加 SimHash 比对

### 4. SimHash 跨 IP 全局去重增强

**问题背景**：之前的 SimHash 去重 key 是 `protocol + "|" + ip`（同 IP 维度），只在同一个 IP 下去重。但实际数据中，不同 IP 上的不同 Host 可能产生相同的 SimHash（如 CDN 默认 404 页面），这些重复结果未被过滤。

**修复方案**：增加跨 IP 全局 SimHash 去重：

```
策略一（保留）：同 IP 维度去重
  key = "protocol|ip" → 海明距离 ≤ 3 则跳过

策略二（新增）：跨 IP 全局去重
  1. 精确匹配：key = "global|{simhash值}" → 完全相同则跳过
  2. 相似匹配：遍历所有已有记录 → 海明距离 ≤ 3 则跳过
```

**日志输出示例**：
```
协议:http://, ip:1.2.3.4, host:a.com 碰撞成功但SimHash与同IP已有结果高度相似(距离≤3),跳过
协议:http://, ip:5.6.7.8, host:b.com 碰撞成功但SimHash与其他IP的已有结果完全相同,跳过
协议:http://, ip:9.10.11.12, host:c.com 碰撞成功但SimHash与其他IP(http://|1.2.3.4)的已有结果高度相似(距离≤3),跳过
```

**涉及文件**：
- `pkg/collision/collision.go` — `collision()` 方法中 SimHash 去重逻辑增加跨 IP 全局去重

```
  → [已有] 自适应分阶段采样

  → [优化] 响应体编码检测与 UTF-8 自动转换（三层防护：meta charset → CJK 字符检查 → GBK 备选解码）
  → [已有] 状态码+内容长度+Server头+Body 联合比对（与直接访问IP结果一致则丢弃）
  → [已有] 内容包含检查、Title 比对、FNV hash 指纹比对、编辑距离相似度比对
  → [优化] 碰撞请求支持最多 3 次重定向跟随（保持 Host 头）
  → [重构] 直接请求 Host 比对（先 DNS 解析验证 → 再严格比对 + SimHash 相似度比对）
  → [已有] SimHash 指纹计算（碰撞成功时计算）
  → [已有] Protocol+IP+Host 全局去重
  → [已有] SimHash 同IP相似内容去重（海明距离 ≤ 3 则丢弃）
  → [增强] SimHash 跨IP全局去重（精确匹配 + 海明距离 ≤ 3 则丢弃）
  → 输出结果（过滤 Invalid 结果）
```
